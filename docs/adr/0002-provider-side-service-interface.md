# ADR-0002: Service интерфейс объявлен provider-side

**Статус:** принято
**Дата:** 2026-02

## Контекст

В каждом модуле есть use case-слой:

```go
// internal/modules/account/internal/service/service.go
type Service interface {
    Create(ctx, ...) (*api.AccountInfo, error)
    GetByID(ctx, id) (*api.AccountInfo, error)
    Transfer(ctx, from, to, amount) error
}

type service struct { /* unexported */ }

func New(...) Service { return &service{...} }
```

Это **противоречит** Go-идиоме «consumer defines a narrow interface»: канон
говорит, что интерфейс надо объявлять там, где он используется
(`handler.Server`, `payment.AccountLookup` и т.п.), а не там, где
реализуется.

## Решение

Service-интерфейс остаётся **рядом с реализацией**. `New` возвращает
интерфейс, не указатель.

Причины — в порядке убывания важности:

1. **Контракт для распила.** Когда модуль уезжает в отдельный сервис, его
   `api.Service` становится gRPC-контрактом. Если интерфейс был узким и
   объявленным у потребителя — у каждого потребителя свой кусочек контракта,
   и собрать единый `.proto` нужно вручную, сводя их вместе. Provider-side
   интерфейс **уже** этот единый контракт.

2. **Один источник правды для тестов.** Все потребители (HTTP handler,
   event handler, in-process adapter) мокают **один** интерфейс. Узкие
   consumer-side интерфейсы породили бы зоопарк моков с разными подмножествами
   методов одной и той же штуки.

3. **`unexported impl + exported interface` = enforced контракт.** Невозможно
   случайно получить доступ к внутренностям реализации обходом интерфейса —
   `service` структура физически недоступна снаружи модуля.

## Альтернативы и почему отказались

**Consumer-side narrow interfaces** — Go-идиома. Хорошо работает в
библиотеках и для тестов одного потребителя. Плохо ложится на сценарий
«позже это уедет в отдельный сервис»: контракт надо будет лепить заново.

**Конкретный тип возвращается из `New`** (`func New(...) *service`).
Простейший вариант, но потребители получают доступ к внутренностям, тесты
неудобные (нельзя замокать), при распиле всё равно придётся объявить интерфейс.

## Последствия

**+** `api/` каждого модуля — готовый контракт для `.proto`-генерации при
распиле.

**+** Меньше моков, меньше зоопарка в тестах.

**−** Интерфейс «шире» канонической Go-рекомендации. Это сознательно.
Если кто-то предложит «давайте перенесём интерфейс в пакет потребителя для
testability» — ответ: нет, причины выше. Зафиксировано здесь, чтобы не
обсуждать заново.
