# ADR-0001: Модульный монолит с вертикальными слайсами

**Статус:** принято
**Дата:** 2026-02

## Контекст

Старт нового продукта. Известно:

- Команда маленькая, операционно микросервисы тянуть нечем (нет SRE, нет
  service mesh).
- Бизнес-домены ещё мигрируют — границы модулей за полгода переедут хотя бы
  раз.
- При успехе через ~год часть модулей надо будет вынести в отдельные сервисы
  (либо по нагрузке, либо по командам).

Варианты, которые рассматривались:

1. **Микросервисы с первого дня.** Дорого операционно; «правильные» границы
   не известны, цена ошибки = миграция БД между сервисами.
2. **Слоистый монолит** (`controllers/`, `services/`, `repositories/`).
   Просто на старте, но к месяцу шестому превращается в шар грязи: код одного
   домена размазан по всем пакетам.
3. **Модульный монолит с вертикальными срезами.** Один процесс, одна БД, но
   код одного домена живёт в одном дереве — и границы между доменами
   обеспечены компилятором.

## Решение

Вертикальные срезы по доменам, изоляция через стандартный механизм Go
`internal/`:

```
internal/modules/<name>/
  api/                публичный контракт (Service, DTOs, events, errors)
  internal/
    domain/           сущности + Repository интерфейс
    repository/       sqlx реализация Repository
    service/          use cases (Service интерфейс + unexported impl)
  module.go           wiring
```

- Чужие модули и `cmd/api/handlers/` могут импортировать **только**
  `<name>/api`.
- `internal/modules/<name>/internal/...` физически недоступен извне модуля —
  Go-компилятор отвергает такой импорт. Это **гарантия**, не соглашение и не
  линтер.
- БД одна, но у каждого модуля своя **схема** (`account.*`, `payment.*`).
  Между схемами **нет FK** — связи по ID, валидация через `api`. Это и есть
  контракт распила.

## Последствия

**+** Границы не дрейфуют: компилятор не даст случайно потянуть `domain` или
`repository` чужого модуля. Никакого `depguard`, никакого ревью.

**+** Распил инкрементальный: модуль можно вынести в отдельный процесс,
поменяв только реализацию его `api.Service` (in-process adapter → gRPC
client). Бизнес-код модуля и потребители не меняются. См. [ADR-0002](0002-provider-side-service-interface.md).

**+** Миграции уже изолированы (`migrations/<module>/`), при выносе схемы в
свою БД — переносим папку.

**−** Один процесс, одна БД → масштабирование вертикальное до момента
распила. Это явный trade-off в пользу скорости разработки.

**−** Тесты модуля поднимают весь монолит, если интеграционные. Лечится
testcontainers + per-module migrations.
