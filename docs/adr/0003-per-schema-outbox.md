# ADR-0003: Transactional outbox per-schema, не общий

**Статус:** принято
**Дата:** 2026-02

## Контекст

Модулям нужны асинхронные побочные эффекты с гарантией доставки (создание
платежа → перевод средств между счетами в другом модуле). Канонический
паттерн — **transactional outbox**: бизнес-write и запись в outbox делаются
в одной транзакции, отдельный relay вычитывает outbox и публикует в брокер.

Вопрос: одна общая таблица `public.outbox` для всего приложения, или
отдельная `<schema>.outbox` для каждого модуля?

## Решение

Каждый модуль владеет своими таблицами: `account.outbox` +
`account.outbox_dead`, `payment.outbox` + `payment.outbox_dead`. Relay
сконфигурирован со списком схем; на каждый цикл — `dispatchBatch` per schema.

Wake-up: один общий `NOTIFY outbox`; relay просыпается на любой публикацию и
последовательно сливает все схемы.

`outbox.Publisher` сконфигурирован на конкретную схему — composition root
создаёт по одному на каждый пишущий модуль:
```go
paymentPublisher := outbox.NewPublisher("payment")
```

## Альтернативы и почему отказались

**Одна общая `public.outbox`** (то, как было сначала):

- **+** Простой relay-loop, один SELECT.
- **−** При выносе модуля в свой сервис надо мигрировать строки этого модуля
  из общей таблицы — невозможно сделать чисто транзакционно.
- **−** Кросс-модульный sequence/coupling: пик в одном модуле забивает
  батч-окно для всех остальных.
- **−** При переходе модуля в свою БД таблица остаётся «у кого-то» — лишняя
  миграция и неочевидное владение.

**Outbox в отдельной схеме `outbox.<module>_events`** (отдельная схема, не
смешано с бизнес-схемой):

- **−** Нет атомарности: бизнес-write в `payment.payments` и outbox-write в
  `outbox.payment_events` — две схемы, одна транзакция (Postgres даст), но
  при распиле бизнес-схема уезжает в новую БД, а outbox остаётся в
  «платформенной». Невозможно поддержать.

Выбран **per-schema**: бизнес-таблицы и outbox владельца живут в одной
схеме, мигрируют вместе.

## Последствия

**+** Распил готов: при выносе модуля его outbox едет с ним. Relay просто
убирает схему из своего списка.

**+** Контракт «модули не лазят в чужие таблицы» соблюдается и для outbox —
каждый Publisher жёстко scoped через `outbox.NewPublisher(schema)`.

**+** Бэклог-метрика и dead-letter рассечены по модулям без extra labels:
гейдж `outbox_backlog{schema="payment"}` сразу показывает, у кого затык.

**−** Relay делает N запросов вместо одного при пробуждении (`drainAll` цикл
по схемам). Для горстки модулей это незначимо; при сотне схем стоит
пересмотреть.

**−** Каждый новый модуль обязан создать `<schema>.outbox` + `outbox_dead`
миграцией. Шаблон-миграция в `migrations/account/20260201000000_outbox.sql`.

## Связанные решения

- **Trace context** хранится в столбце `trace_context jsonb` каждой
  outbox-таблицы — relay восстанавливает span до вызова `Dispatcher.Dispatch`,
  трасса бесшовно продолжается через брокер. См. CLAUDE.md «Tracing».
- **Event ID** propagation: outbox `id` → `amqp.Publishing.MessageId` →
  `messaging.Event.ID` → `<schema>.processed_events`. См. [ADR-0005](0005-consumer-side-idempotency.md).
